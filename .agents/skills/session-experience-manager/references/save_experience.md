# 沉淀会话积累经验 (Save Experience)

当你在当前 Session（会话）中解决了复杂的 Bug、设计了具备挑战性或存在潜在陷阱的架构、踩到了文档未涉及的“坑”，或进行了值得记录的多步试错排查后，**必须**主动将这些经验固化下来，以造福未来的对话或其他智能体。

即使是在收到用户的最终指令或任务被显式标记完成时，你也应当在确认收尾前执行本操作。

## 执行步骤

### A. 维护项目全局信息（强制检查）
无论是否有值得单独写成的“技术踩坑总结”，在当前 session 结束时：
1. 请阅读当前项目根目录下的 `PROJECT_ALL_INFO.md`（如果不存在则新建）。
2. 在文件合适的位置（如无则新建版块，按时间顺序列出），追加当前 session 导致的项目级变更：
   * **需求变更**：新增加了什么能力或修改了哪些核心逻辑。
   * **重要开发变更**：重大的架构调整、模块划分等。
   * **重要技术选型变更**：引入了哪些新的库、API 或关键系统依赖及其原因。
   * *(目的：下个全新 session 进来，只看此表便能直接写代码，不用再问以前选了啥库、之前做过什么。)*

### B. 沉淀单项技术教训（按需执行）
如果在过程中发生了具体的死锁、报错、奇怪的配置陷阱等有价值的“教训”：

#### 1. 总结提炼经验
在内存中（思考过程）梳理刚刚解决的问题：
*   **Context（背景）**：在做什么任务？用的是什么技术栈？
*   **The Problem（坑的具体表现）**：报错是什么？或者不符合预期的现象是什么？
*   **Failed Attempts（我走过的弯路）**：一开始的错误排查思路/错误代码，以及为什么它们行不通。*(可选但强烈推荐写，方便以后的模型理解错误方向)*
*   **The Solution / Key Takeaway（破局之道）**：最终如何解决的？决定性的一行代码、思路转变、或关键的环境变量/前置条件。

#### 2. 写入或追加到具体经验文档
确定该经验所属的维度（例如：`feishu_api_integration`、`google_apps_script_auth`、`remotion_animation_perf` 等）。
*   检查 `${EXPERIENCE_BASE_PATH}/single_experience/` 目录下是否已有相同维度的文档。如有，请使用追加（更新）操作；如无，请新建 Markdown 文件。
*   **命名规范**：文件名请使用英文加下划线，例如 `lottie_to_rive_migration_pitfalls.md`。
*   将你在第1步中梳理的内容，按照清晰的 Markdown 结构（可包含代码块）写入该文件。

#### 3. 更新全局路由表 (Critical)
即使你写入了具体的经验文件，如果没人知道它在哪，它就永远不会被发现。因此，**必须同步**更新全局路由表。
*   利用编辑工具修改或追加到以下文件（若不存在则创建）：
    `${EXPERIENCE_BASE_PATH}/router.md`
*   为其添加一条清晰的索引记录。
    *   **格式要求**：需要贴合当前已有表格。根据问题所属的分类（见 router.md 的现有结构，比如“认证与权限”、“数据类型与格式”等），找到对应的行并**追加相关的关键字**。
    *   **如果是全新的技术范畴**：在表格底部新增一排记录。
    *   不需要把详细解法写进 router.md，只写关键字和对应的文件路径。

## 核心原则

遵循“渐进式披露”理念：
我们不希望所有经验都被写进同一个巨大的文件里（那会撑爆未来会话的 Token）。我们将知识分散在 `single_experience/` 中，并在 `router.md` 中维护它们的“指针”。以后检索时，顺藤摸瓜即可。
