# 检索项目上下文与积累经验 (Retrieve Context & Experience)

当开启一个全新的会话时，或者需要回顾项目组过往的“血泪史”与“避坑指南”时，请严格按照以下步骤进行：

## 执行步骤

1.  **首要步骤（强制执行）：阅读项目全局信息**。
    *   使用工具尝试读取当前项目根目录下的 `PROJECT_ALL_INFO.md`。
    *   这个文件包含了项目至今的所有历史脉络（需求更迭、架构调整、技术选型）。
    *   *不论你只是简单处理 Bug 还是开始全新的功能，在此文件获取宏观认知可以避免你采取重复或冲突的方案。*

2.  **次要步骤（强制执行）：查阅全局路由表**。
    *   读取位于以下组合绝对路径的“避坑”字典（`EXPERIENCE_BASE_PATH` 取自 `SKILL.md` 中定义常量）：
        `${EXPERIENCE_BASE_PATH}/router.md`
    *   这是一个轻量级的目录文件，列出了当前项目所有记录的具体技术踩坑分类以及其对应的文档路径。

3.  **第三步骤（按需执行）：寻找细节排障**。
    *   阅读完 `router.md` 后，仔细比对当前你要解决的复杂问题、报错信息、或任务背景。
    *   如果 `router.md` 中有 **明确且高度相关** 的条目，提取出该条目对应的具体 Markdown 文件路径。
    *   使用 `view_file` 或相关工具，只读取上述找出的具体文件。
    *   **切勿**贪婪地读取 `${EXPERIENCE_BASE_PATH}/single_experience/` 目录下的所有文件。

3.  **应用经验**：
    *   消化具体文件中记载的问题背景（Context）、复现方式（Repro）、错误尝试（Failed Attempts）及最终解法（Solution/Takeaways）。
    *   在当前你的设计方案或修复代码中应用这些解法，避免重蹈覆辙。

## 注意事项

*   若 `router.md` 不存在或为空，说明可能尚未有团队经验沉淀。不要凭空捏造。
*   通过这种“两步走”（先看目录，再看详情）的方式，能最大程度保证相关经验被利用，同时极大程度节约模型上下文的 Token 消耗。此即**渐进式披露 (Progressive Disclosure)** 的核心。
